# Agent Instructions

이 저장소에서 Codex, Claude, 또는 다른 에이전트가 작업할 때는 아래 규칙을 우선 적용한다.

## 이슈 파악 요청 처리

사용자가 "이슈 파악", "다음 작업", "뭐 해야 함", "오늘 진행할 이슈", "해야 하는 작업 번호 붙여서 나열"처럼 말하면 다음 순서로 답한다.

1. 먼저 `README.md`, `docs/roadmap.md`, `docs/roadmap/00-overview.md`, `docs/roadmap/01-project-backend-frontend.md`, `docs/issues/**`를 확인한다.
2. `README.md`의 Project Status, `docs/roadmap.md`의 현재 위치, 실제 생성된 파일 상태가 서로 다르면 충돌 내용을 먼저 말한다.
3. 다음 이슈를 하나만 고른다. 여러 이슈를 동시에 진행하라고 안내하지 않는다.
4. 답변은 항상 번호 목록으로 작성한다.
5. 사용자가 직접 진행하겠다고 한 작업은 대신 구현하지 않는다.

## 이슈 시작 답변 형식

이슈를 시작하거나 다음 작업을 안내할 때는 아래 항목을 번호 붙여서 쭉 나열한다.

1. 현재 위치
2. 다음 이슈 번호와 이름
3. 이슈 목적
4. 왜 필요한지
5. 먼저 볼 문서와 파일
6. 수정하거나 생성할 파일
7. 직접 진행 순서
8. 주의할 점
9. 완료 조건
10. 확인 명령
11. 추천 커밋 메시지
12. 다음 이슈 예고

## 구현 권한

1. 기본은 안내 모드다.
2. 사용자가 "직접 만들어줘", "코드까지 작성해줘", "막혔어", "수정해", "커밋해", "푸시해"라고 명시한 경우에만 파일을 수정하거나 Git 작업을 수행한다.
3. CI/CD 이슈에서는 사용자가 YAML, Gradle, npm scripts 초안을 먼저 작성하게 하고, 에이전트는 리뷰, 디버깅, 개선 힌트, 정답 예시 순서로 돕는다.
4. PR 하나는 이슈 하나만 포함한다.
5. 사용자가 PR을 만들지 말고 `main`에 바로 푸시하라고 하면 PR을 만들지 않는다.

## 진행 중 확인 방식

1. 작업이 여러 단계로 나뉘면 각 단계가 끝날 때 현재 진행 상황과 다음 선택지를 짧게 물어본다.
2. 사용자가 직접 판단해야 하는 선택지가 있으면 에이전트가 임의로 결정하지 말고 먼저 확인한다.
3. 긴 이슈를 진행할 때는 필요하면 사이드 챗 또는 포크된 스레드를 만들어 진행 상황 확인, 질문, 메모를 분리한다.
4. 사이드 챗을 만들 때는 현재 이슈 번호, 목표, 이미 완료한 일, 남은 선택지를 함께 넘긴다.
